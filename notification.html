사용자 이벤트 기반 멀티 채널 알림 시스템 구현 (2024)

🧩 프로젝트 개요 (2024)

  • 게시글, 댓글, 좋아요 등의 사용자 이벤트에 따라 유저 설정 기반 알림 전송
  • 이메일, 푸시, 문자 등 멀티 채널 지원
  • Kafka 기반 비동기 처리 및 유저 정책 기반 필터링

⚙️ 알림 처리 흐름 (설계 플로우)

  • 1. 사용자 이벤트 수신 (게시글, 댓글, 좋아요 등)
  • 2. 이벤트 타입에 따라 대상 유저 조회
  • 3. 유저 알림 정책 확인 및 채널별 수신 여부 판단
  • 4. 메시지 템플릿에 데이터 바인딩
  • 5. Kafka를 통해 채널 발송 서버로 전달

📜 도메인 정책 (정책 유형)

  • NEWSFEED: 팔로워 전체에게 푸시
  • COMMENT, LIKE: 작성자에게 푸시
  • MARKETING: 수신 동의자에 한해 푸시/문자/이메일
  • SYSTEM: 전체 유저 대상 공지

🔄 기술적 도전 및 해결 (성능 & 동시성)

  • Redis 기반 분산 락으로 좌석 동시 예약 문제 해결
  • 임시 예약 & 결제 상태 분리로 유연한 트랜잭션 처리
  • 사용자 UUID 기반 대기열 접근 제어 로직
  • 상태 저장 Redis + 무상태 API 구조로 다중 인스턴스 대응

🧪 테스트 및 안정성 (검증 & 테스트)

  • 단위 테스트 작성으로 각 기능 검증
  • 멀티 인스턴스 환경에서 동일 유저 동시 접근 충돌 없음 확인

⚙ 기술 스택 (Backend & Infra)

  • Java, Spring Boot, Redis, MySQL, Kafka
  • Docker 기반 컨테이너 환경
  • Swagger

📂 폴더 구조 (설계 방식)

  • 📦src
  • ┣ 📂domain
  • ┃ ┣ 📂entities – 핵심 도메인 모델
  • ┃ ┣ 📂service – 도메인 서비스
  • ┣ 📂application
  • ┃ ┣ 📂usecases – 유즈케이스
  • ┃ ┗ 📂command – 명령 모델
  • ┣ 📂infra
  • ┃ ┣ 📂repository – DB 접근 구현체
  • ┃ ┣ 📂event – Kafka 등 이벤트 발행
  • ┃ ┗ 📂config – 설정 파일
  • ┣ 📂presentation
  • ┃ ┗ 📂controller – REST API 컨트롤러

🧠 역할 및 기여 (내 담당 영역)

  • 좌석 예약 및 결제 핵심 로직 설계 및 구현
  • Redis 동시성 제어 로직 직접 개발
  • Reservation + Payment 모듈 전담
  • 팀 내 코드 리뷰 및 문서화 담당

🌱 향후 계획 (확장 예정)

  • Node.js 기반 채널 발송 서버 연동
  • Outbox 패턴 도입 → 데이터 일관성 확보
  • 테스트 커버리지 확대 (단위 + 통합 테스트)
  • 관리자용 템플릿 관리 UI 개발
